Object Persistency the New Way.

 

 

The new way to builds applications is to find out the objects that participate in the business logic. When we finish with objects identification we try to classify them in well defined classes. Finally we build the user interface and connect it with the business objects.  In old style application development we start business model identification and continue with the construction of data base schema. But there is conceptual gap between the database schema and business model. We try to close the gap with a lot of code. With this code we try to bridge the object oriented world with the structural world. We also write code to display the data in user interface and translate the user interaction in data update commands. The result is a hybrid application with unclean building roles. But in the new way there is not structural world. There are only persistent classes.

 

We will try to find out the new way though the construction of application about the family where the parents work in company.

To make a class persistent you must add using directives “using OOAdvantech;”, “using OOAdvantech.MetaDataRepository;” and “using OOAdvantech.PersistenceLayer;” The persistent objects can be run in client process fat client style or application server. The application server is a process like sqlservr.exe. In case of application server you must add the opportunity of remote access to Person class.

Add the code “using system;”,”using OOAdvantech.Remoting;” and “public class Person: MarshalByRefObject, IExtMarshalByRefObject”. Now the objects of class can be run in application server but class isn’t persistent yet. You must add two class attribute and one class field. The attribute for persistency and the attribute for the identity of class. The identity of class it is optional and useful. When you register the assembly in storage the persistency system builds the places to store the objects of class. When you change the person class with additional persisted fields and register again the assembly, the persistency find the place that correspond to the class using the identity of class and make the changes.  If you don’t define the identity of class the persistency system uses the full name of class as identity for example “Family.Person”. There is problem when you change the name of class or namespace for example “Town.Family.Person” the persistency system can’t find the storage place for class Person when you try to update the storage. The persistency system creates a new place and the old place data now are garbage. If you define a Global unique id as identity you eliminate this problem. With Persistent attribute you declare the class persistent. Finally you have to declare a public field “public OOAdvantech.ExtensionProperties Properties;”, the ExtensionProperties field is a bridge between the object and OOAdvnceTech functional area, keeps information about the object and transaction for transaction system, object and storage instance (operative and passive mode) for persistency system etc.

Create object in storage.

Now it’s the time to create an object person in storage “Family”. First of all we open a session with storage “Family” and then we open a system state transition. In the system state transition we call the NewObject function which returns a new object of type Person, set the Name with value “Jack” set the sysStateTransition scoop Consistent property true Now the object person with name “Jack” is persistent. But what are SystemStateTransition, and ObjectStateTransition? Some words about Transaction System. Transition System is a feature of OOAdvanceTech library. When we initiate a state transition automatically the system begin a transaction, all changes run under transaction. If something goes wrong and an exception raised the system return to the previous state. For more information you can read the transactions topic.

 

 

private void NewObject_Click(object sender, System.EventArgs e)

{

 

        ObjectStorage objectStorage = ObjectStorage.OpenStorage("Family",

                            "localhost",

                            "OOAdvantech.MSSQLPersistenceRunTime.StorageProvider");

 

        using (SystemStateTransition sysStateTransition = new SystemStateTransition())

        {

            Family.Person person = objectStorage.NewObject<Family.Person>();

            person.Name = "Jack";

            sysStateTransition.Consistent = true;

        }

}

We have to change the code of Person class as following. The first thing that changes is persistency declaration. We have change PersistentMember(255) declaration to PersistentMember(255,"_Name"). The persistent attribute has an additional parameter, the name o field that contains the actual data. In property setter we add an Object State Transition Scoop. There are two cases.

·           The case where there isn’t Transaction in Logical call context, system initiates transaction when execution cursor goes into object state transition scoop and commits transaction when execution cursor goes out the scoop. After the transaction commit the value o property is persistent.

·           The case where there is Transaction in Logical call context, system doesn’t initiate transaction but enlist object change in logical call context transaction. When Logical call context transaction committed the value o property will be persistent.

Now we can add persistent relationship between class Person and Address. The name of relationship is “LiveAt”. The relationship has the characteristic of one way or two way access. The two ways access means that we have the ability to access the address of person from person and the person that live in a specific address from address object. For two way access we must define property or field for relationship at both classes. In our case the relationship is one way access or with UML terminology navigable to address. We define a property with name Address at Person class. And we add Association attribute. Some words about Association attribute. The first parameter of Association is the name of relationship the next is the type of other and class, the next is the Role of AssociationEnd (the property is AssociationEnd in UML terminology). After that we add PersistentMember attribute with first parameter the persistence flag in this case tells to persistency that relationship has ReferentialIntegrity. This means if at least one person lives at address then this address can’t be deleted from object storage. To delete we must remove all relationships between the address and persons that live at this address. For more information look at PersistencyFlag reference manual. The last parameter is the implementation field name. With RoleAMultiplicityRange we define the multiplicity of address this means that persons has to has one address. With RoleAMultiplicityRange we define the multiplicity of person this means in one address can live many persons.

 

public class Person:MarshalByRefObject,IExtMarshalByRefObject

{

 

  private Address _Address;

  [Association("LiveAt", typeof(Family.Address), Roles.RoleA, "DD8799B6-212A-4d25-AA09-CA3F0E6AE7DD")]   [PersistentMember(PersistencyFlag.ReferentialIntegrity,"_Address")]

  [RoleAMultiplicityRange(1,1)]

  [RoleBMultiplicityRange(0)]

  public Address Address

  {

     get

     {

       return _Address;

     }

     set

     {

       if(_Address!=value)

       {

         using( ObjectStateTransition objStateTransition=new ObjectStateTransition(this))

         {

             _Address=value;

             objStateTransition.SetComplete();

         }

       }

Inheritance.

Inheritance relationship is something new in persistency systems but very popular in programming languages. An example is the Person and Employee. The employee has Salary.

 

using OOAdvantech.MetaDataRepository;

using OOAdvantech.Transactions;

namespace Family

{

  [BackwardCompatibilityID("{E9D67233-E834-45FE-8246-A772E0698E2D}")]

  [Persistent("<ExtMetaData><RDBMSInheritanceMapping>OneTablePerConcreteClass</RDBMSInheritanceMapping></ExtMetaData>")]

  public class Employee:Person

  {

     private double _Salary;

     [BackwardCompatibilityID("+1")]

     [PersistentMember("_Salary")]

     public double Salary

     {

       get

       {

         return _Salary;

       }

       set

       {

         if(_Salary!=value)

         {

             using( ObjectStateTransition objStateTransition=new ObjectStateTransition(this))

             {

                  _Salary=value;

                  objStateTransition.SetComplete();

             }

         }

       }

     }

 

     public Employee()

     {

    

     }

     public Employee(string name, double salary): base( name)

     {

       _Salary=salary;

     }

  }

}

 

 

There is something deferent from other persistent class declaration. We additional declare extension meta data for inheritance. In RDBMS subsystem the classes mapped to tables and persistent members to columns relationships to foreign keys etc. All that implicitly. Some times like inheritance we have more than one mapping choice. If we not declare extension meta data for inheritance the persistency system choose the default option. In our case we declare OneTablePerConcreteClass. This choice it is not the default. The RDBMS mapping model is out of scoop of this article. Association Class.

The association class used when we want information for class relationship. For example we have the employee which works in company. The employee has relationship with the company that work. But it isn’t enough. We want to know the job of employee in company. The job is association class and keeps information about the job of employee.

 

 

 

The Name property keeps the name of job, the start date etc. We have to add code for two classes. Firstly we add code for class company and then code for relationship between company and employee.

 

[BackwardCompatibilityID("{0DAA518F-51BE-4AF6-A3FE-520BDCB798D1}")]

[Persistent]

public class Company:MarshalByRefObject,IExtMarshalByRefObject

{

  public OOAdvantech.ExtensionProperties Properties;

 

 

  private Set _Employees=new Set();

 

  [BackwardCompatibilityID("+1")]

  [Association("Job",typeof(Family.Employee),Roles.RoleB)]

  [PersistentMember("_Employees")]

  [RoleBMultiplicityRange(0)]

  [AssociationClass(typeof(Family.Job))]

  public Set Employees

  {

    get

    {

      return new Set(_Employees);

    }

  }

  public void AddEmployee(Job job)

  {

    if(job==null)

      return;

   

    if(job.Employee==null)

      throw new System.Exception("the member Employee of job isn't set");

 

    using(ObjectStateTransition stateTransition=new ObjectStateTransition(this))

    {

      _Employees.Add(job);

      stateTransition.Consistent=true;

    }

  }

  public void RemoveEmployee(Job job)

  {

    if(job==null)

      return;

 

    using(ObjectStateTransition stateTransition=new ObjectStateTransition(this))

    {

      _Employees.Remove(job);

      stateTransition.Consistent=true;

    }

  }

  private string _Name;

 

  [BackwardCompatibilityID("+2")]

  [PersistentMember("_Name")]

  public string Name

  {

    get

    {

      return _Name;

    }

    set

    {

      if(_Name!=value)

      {

        using(ObjectStateTransition objStateTransition=new ObjectStateTransition(this))

        {

           _Name=value;

           objStateTransition.Consistent=true;

        }

      }

    }

  }

}

 

The code is the same as the person class code the attribute of relationship property Employees is approximately the same with the attribute of Childrens but Employees has one attribute more, the [AssociationClass(typeof(Family.Job))]. This attribute declare that the relationship has association class. Another new in this relationship is that the collection doesn’t contain employee objects but contains job objects. With other words we can tell that contains association class object. The association class is the Job. The Job class has one more attribute, the “[AssociationClass(typeof(Family.Company),typeof(Family.Employee),"Job")]”. The AssociationClass attribute constructor takes three parameter, the first is the role A class type the second the role B class type and third the name of relationship. Also has two members for the related object in this case the Employee and Employer, to declare a member of association class as related object member you must define on them AssociationClassRole attribute with parameter the type of role (RoleA or RoleB) and optional the implementation field if the member is property. If we create a job object set the Employee member with employee object, and adds it to company object with the AddEmployee method then at the end of state transition we will have in Employees collection the job object and two association role member of job with correct values employee and company objects. We must mention that we have added the Employers collection in employee class with the same characteristics with Employees collection of Company class.

Conclusion.

With OOAdvanceTech persistency system you don’t need write code for data base like create tables, store procedure mapping the object oriented world to tables. The persistency system takes care for that. The only that you needed to do is to write classes and declare that persistent. Also persistency gives you a powerful and friendly query language to retrieve object or data or both from object storage. Finally the persistency system takes care to store the objects in deferent data base like Microsoft SQL Server, Oracle or deferent technology storage media like XML files with no impact on your code.
